# 让龙虾写出高质量软件：需求分析 → 四维检查 → 测试验证

## 前言

"虾虾，帮我写一个用户认证模块。"

龙虾开始敲代码，三分钟后：

```typescript
function login(username: string, password: string) {
  return db.query(`SELECT * FROM users WHERE username='${username}' AND password='${password}'`);
}
```

代码能跑，但一眼望去全是坑：
- **安全漏洞**：SQL 注入
- **可靠性问题**：无错误处理
- **性能隐患**：无索引优化
- **易用性缺失**：无友好提示

这不是龙虾不够聪明，而是 **缺少系统化的质量保障流程**。

---

## 阶段一：需求分析

### 为什么需要需求分析？

**没有需求分析，龙虾只能"猜"你要什么。**

你说"写个登录功能"，龙虾理解的是：
- 最简单的用户名密码校验？
- 还是支持 OAuth、短信验证码、扫码登录？
- 需要记住登录状态吗？
- 密码强度有要求吗？
- 登录失败次数限制？

### 如何让龙虾做需求分析？

**Prompt 模板**：

```
我要实现一个【功能名称】。

请帮我做需求分析：
1. 这个功能的核心目标是什么？
2. 需要哪些子功能？
3. 有哪些边界情况需要考虑？
4. 有哪些安全风险？
5. 验收标准是什么？

请输出结构化的需求文档。
```

### 实际案例：用户认证模块

**用户输入**：
```
我要实现一个用户认证模块。

请帮我做需求分析：
1. 这个功能的核心目标是什么？
2. 需要哪些子功能？
3. 有哪些边界情况需要考虑？
4. 有哪些安全风险？
5. 验收标准是什么？
```

**龙虾输出**：

```markdown
# 用户认证模块需求分析

## 1. 核心目标
提供安全、可靠的用户身份验证机制，支持多种登录方式。

## 2. 子功能清单
| 功能 | 优先级 | 说明 |
|------|--------|------|
| 用户名密码登录 | P0 | 基础登录方式 |
| 密码加密存储 | P0 | bcrypt，cost=12 |
| 登录状态管理 | P0 | JWT + Refresh Token |
| 登录失败限制 | P1 | 5 次失败锁定 15 分钟 |
| 密码强度校验 | P1 | 至少 8 位，含数字和字母 |
| OAuth 登录 | P2 | 微信、GitHub |

## 3. 边界情况
- 用户名不存在 → 提示"用户名或密码错误"（不泄露信息）
- 密码错误 5 次 → 锁定账户
- Token 过期 → 自动刷新
- 并发登录 → 踢掉旧会话

## 4. 安全风险
| 风险 | 防护措施 |
|------|----------|
| SQL 注入 | 参数化查询 |
| 暴力破解 | 登录失败限制 + 验证码 |
| 中间人攻击 | HTTPS + Secure Cookie |
| 密码泄露 | bcrypt 加密 + 盐值 |

## 5. 验收标准
- [ ] 正确的用户名密码能登录成功
- [ ] 错误密码提示"用户名或密码错误"
- [ ] 连续 5 次失败锁定账户
- [ ] Token 有效期 2 小时，Refresh Token 7 天
- [ ] 密码存储使用 bcrypt，cost=12
```

**关键点**：需求分析阶段，让龙虾把"模糊的需求"变成"可验证的清单"。

## 阶段二：四维检查

### 检查一：可靠性

**核心问题**：代码在异常情况下能正常工作吗？

**检查清单**：

| 检查项 | 说明 | 示例 |
|--------|------|------|
| 错误处理 | 是否捕获所有可能的异常？ | `try-catch` 包裹数据库操作 |
| 边界条件 | 空值、超长输入、特殊字符 | 用户名为空、密码 1000 字符 |
| 重试机制 | 网络失败是否重试？ | 数据库连接失败重试 3 次 |
| 降级方案 | 主流程失败有备选吗？ | OAuth 失败回退到密码登录 |
| 幂等性 | 重复请求会出问题吗？ | 重复提交登录表单 |

### 四维检查实战

**原始代码**：

```typescript
function login(username: string, password: string) {
  return db.query(`SELECT * FROM users WHERE username='${username}' AND password='${password}'`);
}
```

**龙虾四维检查报告**：

```markdown
# 用户认证模块四维检查报告

## 1. 可靠性 ❌

**问题**：
- 无错误处理：数据库连接失败会抛出未捕获异常
- 无边界检查：用户名密码为空会导致查询失败
- 无重试机制：网络抖动直接失败

**修复建议**：
```typescript
async function login(username: string, password: string) {
  // 边界检查
  if (!username || !password) {
    throw new Error('用户名和密码不能为空');
  }
  
  // 错误处理 + 重试
  const maxRetries = 3;
  for (let i = 0; i < maxRetries; i++) {
    try {
      const user = await db.query(
        'SELECT * FROM users WHERE username = ?',
        [username]
      );
      // ...
    } catch (error) {
      if (i === maxRetries - 1) throw error;
      await sleep(1000 * Math.pow(2, i)); // 指数退避
    }
  }
}
```

## 2. 性能 ⚠️

**问题**：
- 无索引：`username` 字段无索引，查询慢
- 无缓存：每次都查数据库
- 无连接池：每次新建连接

**优化建议**：
- 添加索引：`CREATE INDEX idx_username ON users(username)`
- 添加缓存：用户信息缓存到 Redis
- 使用连接池：`mysql.createPool()`

## 3. 安全 ❌

**问题**：
- **SQL 注入**：直接拼接 SQL，危险！
- 密码明文：数据库存储明文密码
- 无登录限制：可无限暴力破解

**修复建议**：
```typescript
// 参数化查询
const user = await db.query(
  'SELECT * FROM users WHERE username = ?',
  [username]
);

// 密码校验
const isValid = await bcrypt.compare(password, user.password_hash);

// 登录限制
const failCount = await redis.get(`login_fail:${username}`);
if (failCount >= 5) {
  throw new Error('账户已锁定，请 15 分钟后重试');
}
```

## 4. 易用性 ⚠️

**问题**：
- 无类型定义：参数类型不明确
- 无错误提示：失败无友好提示
- 无文档注释：其他开发者不知道怎么用

**改进建议**：
```typescript
interface LoginRequest {
  username: string;
  password: string;
}

interface LoginResponse {
  success: boolean;
  token?: string;
  message: string;
}

/**
 * 用户登录
 * @param request 登录请求
 * @returns 登录结果
 * @throws {Error} 用户名或密码错误
 * @example
 * const result = await login({ username: 'admin', password: '123456' });
 */
async function login(request: LoginRequest): Promise<LoginResponse> {
  // ...
}
```

## 总结

| 维度 | 评分 | 主要问题 |
|------|------|----------|
| 可靠性 | ❌ 不合格 | 无错误处理、无边界检查 |
| 性能 | ⚠️ 待改进 | 无索引、无缓存 |
| 安全 | ❌ 不合格 | SQL 注入、密码明文 |
| 易用性 | ⚠️ 待改进 | 无类型定义、无文档 |

**建议**：修复安全和可靠性问题后再上线。
```

    it('正确登录应返回 token', async () => {
      const res = await request(app)
        .post('/auth/login')
        .send({ username: 'admin', password: 'Admin123' });

    it('密码错误应返回 401', async () => {

        .post('/auth/login')

      expect(res.status).toBe(401);
    });
  });
});

## 最佳实践总结

### 1. 需求分析阶段

```markdown
✅ 必须做：
- 让龙虾输出结构化需求文档
- 明确验收标准
- 识别边界情况和安全风险

❌ 不要做：
- 只说"写个 XX 功能"
- 不确认需求就动手
- 没有验收标准
```

### 2. 四维检查阶段

```markdown
✅ 必须做：
- 每个功能都进行四维检查
- 记录检查结果和修复建议
- 修复后再进入下一阶段

❌ 不要做：
- 只检查功能是否实现
- 忽略安全和性能
- 检查完不改
```

### 3. 测试验证阶段

```markdown
✅ 必须做：
- 编写完整的测试计划
- 覆盖正常、异常、安全场景
- 自动化测试用例

❌ 不要做：
- 只测试正常流程
- 没有安全测试
- 手动测试不自动化
```
