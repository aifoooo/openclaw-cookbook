# AI 编程失败教训：一次痛苦的调试经历

> 这是一篇迟到的复盘。记录了我在使用 AI（OpenClaw）开发 openclaw-monitor 项目时遇到的挫折，以及对失败原因的反思。出路部分只是初步想法，还需要后续探索。

---

## 一、事情经过

### 期望

我以为配置好 OpenClaw，它就能按照我的预期写出满意的代码。一个简单的监控界面，应该几天就能搞定。

### 现实

调试过程中，问题一遍遍地解，却始终没有彻底解决。有时候改好了，过一会儿又出问题了。交互细节调了又调，效果始终不如意。

最后焦头烂额，问题依然存在。

## 二、失败案例

### 案例 1：滚动问题（四次提交都没解决）

**问题**：消息列表滚动行为不符合预期——用户滚动到中间查看历史，新消息到来时会被自动滚动到底部，打断阅读。

**提交记录**：

| 提交 | 描述 |
|------|------|
| `3a8fca7` | feat: 优化聊天列表刷新和消息滚动逻辑 |
| `85063e1` | fix: 修复消息滚动逻辑 |
| `b9eafae` | fix: 彻底修复消息滚动问题 |
| `907c1f0` | fix: 完全禁用消息自动滚动，解决滚动和闪烁问题 |

四次提交，全部围绕同一个问题。

**最终解决方案**：完全禁用自动滚动，改为全部手动控制。

**反思**：为什么同一个问题需要四次提交？这四次提交之间有什么区别和联系？当时的我是否真的理解了问题的根因？

### 案例 2：Vue watch 导致重复请求

**问题**：点击一次聊天，触发两次 API 请求，导致消息内容重复显示。

**排查过程**：

1. **测试脚本失效**：统计显示 54 次请求，手动测试只有 1-2 次。原因是 `journalctl --vacuum-time=1s` 并没有真正清空日志，统计的是历史累积请求。

2. **测试环境干扰**：文件监视器、自动同步、WebSocket 心跳等都在产生请求，干扰了统计。

3. **Vue 生命周期理解不足**：

```typescript
// watch + immediate: true + onMounted 的组合导致重复触发
watch(() => props.chatId, () => {
  loadMessages();
}, { immediate: true });  // 组件创建时立即执行

onMounted(() => {
  loadMessages();  // 组件挂载时又执行
});
```

**最终解决方案**：移除 `immediate: true`，只在 `onMounted` 中加载。

**反思**：如果一开始就对 Vue 生命周期有清晰认识，是否能避免这个问题？

### 案例 3：WebSocket 连接问题

**问题**：WebSocket 连接失败，但后端日志显示正常。

**现象**：
- 前端显示 "WebSocket 连接: ❌ 未连接"
- 后端日志显示连接成功
- API 请求正常

**排查过程**：一直在检查前端代码、WebSocket 配置、代理设置...却没有先确认后端日志是否真的显示连接成功。

**反思**：花了大量时间排查前端，却忽视了先验证后端日志这个简单步骤。

## 三、失败原因剖析

### 原因 1：过度依赖 AI，忽视了基础能力

**现象**：
- 遇到 Vue 生命周期问题，需要查文档才能确认执行顺序
- 遇到测试脚本问题，不确定 `--vacuum-time` 的实际行为
- 遇到 WebSocket 问题，不确定如何验证连接是否成功

**分析**：当 AI 给出建议后，我缺乏快速验证的能力。AI 说什么就信什么，没有批判性思维。

**教训**：AI 是助手，不是老师。你仍然需要具备扎实的基础知识，才能判断 AI 的建议是否正确。

### 原因 2：缺乏系统性调试方法

**现象**：
- WebSocket 问题：一直在改前端配置，却没有先验证后端日志
- 滚动问题：四次提交，每次都是"尝试性"修改，不确定是否有效
- 测试脚本：设计缺陷导致误判，却没有发现

**分析**：没有形成系统性的调试方法论。遇到问题就开始改代码，而不是先收集信息、形成假设、验证假设。

**教训**：调试需要方法论。不是"try random things"，而是"hypothesis → test → learn"。

### 原因 3：上下文丢失，重复踩坑

**现象**：
- 滚动问题：在 3a8fca7 中已经"优化"过为什么在 907c1f0 中又作为新问题处理？
- Vue watch 问题：排查文档写得很详细，但下次遇到类似问题是否还会踩坑？

**分析**：每次会话都是新的开始，之前的调试经验没有积累下来。这次踩的坑，下次可能还会踩。

**教训**：需要建立个人知识库，记录踩坑经历和解决方案。

### 原因 4：人机协作模式不当

**现象**：
- AI 说什么就改什么
- 没有主动提供上下文（后端日志、测试结果）
- 期望 AI 能自动解决所有问题

**分析**：把 AI 当成了能读懂人心的神仙，而不是需要清晰指令的工具。

**教训**：人机协作需要正确分工。人负责定义问题、提供上下文、验证结果；AI 负责生成代码、执行任务。

## 四、出路（初步想法）

> 这部分只是初步想法，还需要后续探索补齐。

### 方向 1：建立调试 checklist

遇到问题时，按照固定流程排查：
1. 收集信息（日志、配置、现象）
2. 形成假设
3. 验证假设
4. 修复问题
5. 确认修复

### 方向 2：积累踩坑记录

建立个人知识库，记录：
- 问题现象
- 排查过程
- 解决方案
- 相关原理

### 方向 3：明确人机分工

- 人：定义问题、提供上下文、验证结果
- AI：生成代码、执行任务、提供建议

### 方向 4：学习 AI 调试工具

- 了解 AI 的能力边界
- 掌握如何给 AI 提供有效的上下文
- 学会设计有效的提示词

---

## 五、总结

这次失败的教训：

1. **AI 是助手，不是老师**：你仍然需要扎实的基础知识
2. **调试需要方法论**：不是 try random things，而是 hypothesis → test → learn
3. **上下文比答案重要**：学会提供有效的上下文，比获得答案更有价值
4. **人机协作需要分工**：明确各自的职责，才能发挥最大价值

**最后的感想**：AI 编程确实能提高效率，但如果期望过高、方法不当，反而会降低效率。这次经历让我意识到，最重要的不是学习 AI 工具，而是重新审视自己的基础能力和方法论。
