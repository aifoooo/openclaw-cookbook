# 永不停止的龙虾：让 OpenClaw 不间断工作直到目标完成

## 前言

"虾虾，帮我写一个完整的项目。" 你满怀期待地发出指令。

三小时后，你回来看进度——

**"连接超时，请稍后重试。"**

或者更糟：

**"任务已完成！"** 

但你打开代码，发现只有一堆半截的函数和注释 `// TODO: implement this`。

这不是模型不够聪明，而是 **Agent 的"记忆"断了**。

---

## 问题本质：为什么 Agent 会"失忆"？

### 现实很残酷

| 问题 | 表现 | 根因 |
|------|------|------|
| **Session 重置** | 每次对话都是新开始 | 无状态持久化 |
| **上下文溢出** | 长任务中途截断 | 模型上下文窗口有限 |
| **网络抖动** | 随机失败 | 无重试机制 |
| **模型降级** | 主模型不可用 | 无 fallback 链 |
| **过早宣布完成** | 留下半成品 | 无进度追踪 |

### 核心矛盾

**模型每一轮都在有限上下文中工作，每次执行都是一次"记忆重置"。**

Anthropic 的研究指出，长时任务容易出现两类问题：

1. **一口气写到爆**：模型大规模编码，直到上下文耗尽，留下"半截代码"
2. **过早宣布完成**：看到一些 UI 或 API 就误以为功能齐全

---

## 解决方案：四大支柱

### 支柱一：状态持久化

**让 Agent 有"记忆"**

#### OpenClaw 的 Session 机制

OpenClaw 通过 `.jsonl` 文件持久化所有对话状态：

```
/root/.openclaw/agents/{agentId}/sessions/{sessionId}.jsonl
```

每一行是一个 JSON 对象，记录完整的消息历史和状态变更：

```json
{"type":"session","version":3,"id":"7add3c20-...","timestamp":"2026-03-15T04:00:01Z"}
{"type":"model_change","provider":"tencentcodingplan","modelId":"glm-5"}
{"type":"message","role":"user","content":"帮我写一个项目"}
{"type":"message","role":"assistant","content":"好的，我来分析需求..."}
```

**关键特性**：
- **自动恢复**：下次对话自动加载历史
- **状态追踪**：记录模型切换、thinking 级别等
- **跨会话连续**：即使重启 Gateway，对话不丢失

#### OpenClaw 的记忆系统

OpenClaw 通过 `AGENTS.md` 定义每次会话自动加载的"记忆"：

```markdown
# AGENTS.md
## Every Session
1. Read `SOUL.md` — this is who you are
2. Read `USER.md` — this is who you're helping
3. Read `memory/YYYY-MM-DD.md` — recent context
4. 查询最近 10 条记忆 — user preferences
5. If in MAIN SESSION: Also read `MEMORY.md`
```

**三层记忆架构**：

| 层级 | 文件 | 用途 |
|------|------|------|
| **短期** | `sessions/*.jsonl` | 当前对话上下文 |
| **中期** | `memory/YYYY-MM-DD.md` | 每日笔记 |
| **长期** | `MEMORY.md` | 永久记忆 |

**实际效果**：每次会话开始，Agent 先"恢复记忆"，再继续工作。就像人类每天早上看日程表一样。

### 支柱二：错误恢复机制

**让 Agent 能"自愈"**

#### OpenClaw 的模型降级链

在 `openclaw.json` 中配置 fallback 链：

```json
{
  "agents": {
    "mime-qq": {
      "model": "tencentcodingplan/glm-5",
      "fallbacks": [
        "tencentcodingplan/hunyuan-2.0-thinking",
        "deepseek/deepseek-reasoner"
      ]
    }
  }
}
```

**工作原理**：
1. 主模型 `glm-5` 失败 → 自动切换到 `hunyuan-2.0-thinking`
2. `hunyuan` 也失败 → 切换到 `deepseek-reasoner`
3. 全部失败 → 返回错误，但至少尝试过

**实际案例**：腾讯云 API 偶发网络抖动（约 10 次/3 天），OpenClaw 自动降级到备用模型，用户无感知。

#### 上下文管理策略

OpenClaw 配置了智能的上下文管理：

```json
{
  "contextPruning": {
    "mode": "cache-ttl",
    "ttl": "1h",
    "keepLastAssistants": 3,
    "softTrimRatio": 0.3,
    "hardClearRatio": 0.5
  },
  "compaction": {
    "mode": "safeguard",
    "reserveTokens": 30000
  }
}
```

**策略说明**：
- **cache-ttl**：缓存 1 小时内的上下文
- **keepLastAssistants**：保留最近 3 轮助手回复
- **softTrimRatio**：软裁剪 30% 旧内容
- **reserveTokens**：预留 30000 tokens 安全边界

**效果**：长任务不会因为上下文溢出而中断，系统自动管理上下文窗口。

### 支柱三：监督者架构

**让 Agent 有"指挥"**

Anthropic 提出的双 Agent 架构，核心思想是 **规划与执行分离**：

```
┌─────────────────────────────────────────────────┐
│              Initializer Agent                   │
│  （首席架构师 - 只运行一次）                      │
│                                                  │
│  1. 需求分解 → JSON 结构化功能清单               │
│  2. 创建 progress 文件记录进度                   │
│  3. 生成 init.sh 环境启动脚本                    │
└─────────────────────────────────────────────────┘
                      ↓
┌─────────────────────────────────────────────────┐
│              Coding Agent                        │
│  （开发者 - 循环迭代）                            │
│                                                  │
│  每轮开始：                                       │
│  1. 检查目录、读 git 记录、跑 init.sh            │
│  2. 从清单选一个未完成功能                       │
│  3. 执行并验证                                   │
│  4. 通过才标记完成并 commit                      │
└─────────────────────────────────────────────────┘
```

#### OpenClaw 的实践

OpenClaw 通过 `MEMORY.md` 实现类似效果：

```markdown
# MEMORY.md - 长期记忆

## 当前进行中的项目

### openclaw-cookbook
- **状态**：进行中
- **已完成**：龙虾初相遇、深度调研、长期的伙伴
- **下一步**：写一篇关于 Agent 可靠性的文章

### 犯过的错（不再犯）
- 2026-03-13：成本表述错误
- 教训：成本/数据类陈述，必须拆分每个组件逐一核实
```

**核心思想**：
- 每次会话开始，先读取 `MEMORY.md` 恢复上下文
- 记录当前进度、下一步计划、历史教训
- 新会话自动"继承"之前的工作

**实际效果**：即使会话中断，下次启动时 Agent 知道"我在做什么"、"下一步该做什么"、"什么坑要避开"。

### 支柱四：主动监控机制

**让 Agent 有"自觉"**

#### OpenClaw 的 Heartbeat 机制

OpenClaw 支持定期心跳检查，通过 `HEARTBEAT.md` 配置：

```markdown
# HEARTBEAT.md

# 定期检查任务（每 30 分钟）
- 检查邮件：是否有紧急消息
- 检查日历：24 小时内是否有重要事件
- 检查天气：是否需要提醒带伞

# 如果一切正常，回复 HEARTBEAT_OK
# 如果有需要关注的事项，主动通知用户
```

**工作原理**：
1. OpenClaw Gateway 定期发送心跳提示
2. Agent 读取 `HEARTBEAT.md`，执行检查任务
3. 无事项 → 回复 `HEARTBEAT_OK`（静默）
4. 有事项 → 主动通知用户

**实际效果**：Agent 不只是被动响应，而是"主动关心"用户的需求。

#### OpenClaw 的 Cron 定时任务

通过 `qqbot-cron` skill，OpenClaw 支持定时提醒：

```bash
# 创建一次性提醒
/remind 2026-03-17 09:00 记得开会

# 创建周期性提醒
/remind daily 09:00 每日站会

# 查看所有提醒
/remind list
```

**应用场景**：
- 定时任务提醒
- 周期性工作检查
- 重要事件预警

## 后记

让 Agent 不间断工作，本质上是 **软件工程最佳实践的深度融入**：

- **持久化** → Session 文件 + 三层记忆
- **容错设计** → 模型降级链 + 上下文管理
- **状态管理** → MEMORY.md + 每日笔记
- **主动监控** → Heartbeat + Cron

OpenClaw 已经实现了大部分基础设施。我们只需要：

1. **用好 AGENTS.md** — 定义每次会话的启动流程
2. **维护 MEMORY.md** — 记录长期状态和进度
3. **配置 fallbacks** — 确保模型可用性
4. **善用 Heartbeat** — 让 Agent 主动关心

未来的竞争焦点，将从模型能力转向 **Agent 工程化能力**。

谁能更有效地构建出可靠、可扩展、可维护的 Agent 系统，谁就能真正释放 AI Agent 的自动化潜力。
