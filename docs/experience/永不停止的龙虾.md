# 永不停止的龙虾：让 OpenClaw 不间断工作直到目标完成

## 前言

"虾虾，帮我写一个完整的项目。" 你满怀期待地发出指令。

三小时后，你回来看进度——

**"连接超时，请稍后重试。"**

或者更糟：

**"任务已完成！"** 

但你打开代码，发现只有一堆半截的函数和注释 `// TODO: implement this`。

这不是模型不够聪明，而是 **Agent 的"记忆"断了**。

---

## 问题本质：为什么 Agent 会"失忆"？

### 现实很残酷

| 问题 | 表现 | 根因 |
|------|------|------|
| **Session 重置** | 每次对话都是新开始 | 无状态持久化 |
| **上下文溢出** | 长任务中途截断 | 模型上下文窗口有限 |
| **网络抖动** | 随机失败 | 无重试机制 |
| **模型降级** | 主模型不可用 | 无 fallback 链 |
| **过早宣布完成** | 留下半成品 | 无进度追踪 |

### 核心矛盾

**模型每一轮都在有限上下文中工作，每次执行都是一次"记忆重置"。**

Anthropic 的研究指出，长时任务容易出现两类问题：

1. **一口气写到爆**：模型大规模编码，直到上下文耗尽，留下"半截代码"
2. **过早宣布完成**：看到一些 UI 或 API 就误以为功能齐全

---

## 解决方案：四大支柱

### 支柱一：状态持久化

**让 Agent 有"记忆"**

#### OpenClaw 的实践

OpenClaw 通过 session 文件记录所有对话：

```
/root/.openclaw/agents/{agentId}/sessions/{sessionId}.jsonl
```

每一行是一个 JSON 对象，记录了完整的消息历史：

```json
{"type":"message","timestamp":"2026-03-16T10:00:00Z","message":{"role":"user","content":"帮我写一个项目"}}
{"type":"message","timestamp":"2026-03-16T10:00:05Z","message":{"role":"assistant","content":"好的，我来分析需求..."}}
```

#### 进阶：使用 LangGraph 的 Checkpoint

```python
from langgraph.checkpoint.sqlite import SqliteSaver

# 配置持久化存储
checkpointer = SqliteSaver("agent_state.db")

# 使用 thread_id 管理会话
config = {"configurable": {"thread_id": "project-xyz"}}

# 即使中断，下次可以从断点恢复
result = graph.invoke(input, config=config)
```

**关键点**：
- 使用 `thread_id` 标识不同任务
- 状态自动保存到数据库
- 中断后可以从断点恢复

### 支柱二：错误恢复机制

**让 Agent 能"自愈"**

#### 分层错误处理

| 层级 | 错误类型 | 处理策略 |
|------|----------|----------|
| **工具层** | API 超时、格式错误 | 重试、回退 |
| **Agent 层** | 推理错误、幻觉 | 反思、纠正 |
| **系统层** | 网络故障、模型不可用 | 降级、告警 |

#### OpenClaw Monitor 的实践

我们为代理添加了三层保护：

```typescript
// 1. 错误重试
const MAX_RETRIES = 3;
const RETRY_DELAY = 1000;

function forwardRequest() {
  proxy.web(req, res, options, (error) => {
    if (shouldRetry(error) && retryCount < MAX_RETRIES) {
      setTimeout(() => forwardRequest(), RETRY_DELAY);
    } else {
      // 记录错误并返回
      logError(error);
      res.status(502).json({ error: 'Proxy Error' });
    }
  });
}

// 2. 模型降级链
const FALLBACK_CHAIN = [
  'tencentcodingplan/glm-5',
  'tencentcodingplan/hunyuan-2.0-thinking',
  'deepseek/deepseek-reasoner'
];

// 3. 健康检查
setInterval(() => {
  checkUpstreamHealth();
}, 30000);
```

#### 可重试的错误类型

```typescript
function isRetryableError(error: Error): boolean {
  const retryableMessages = [
    'ECONNRESET',      // 连接重置
    'ETIMEDOUT',       // 超时
    'ENOTFOUND',       // DNS 解析失败
    'ECONNREFUSED',    // 连接被拒绝
    'socket hang up',  // Socket 断开
    'network',         // 网络错误
    'timeout',         // 超时
  ];
  
  return retryableMessages.some(msg => 
    error.message.toLowerCase().includes(msg.toLowerCase())
  );
}
```

### 支柱三：双 Agent 架构

**让 Agent 有"指挥"**

Anthropic 提出的双 Agent 架构，解决了长时任务的两大痛点：

```
┌─────────────────────────────────────────────────┐
│              Initializer Agent                   │
│  （首席架构师 - 只运行一次）                      │
│                                                  │
│  1. 需求分解 → JSON 结构化功能清单               │
│  2. 创建 progress 文件记录进度                   │
│  3. 生成 init.sh 环境启动脚本                    │
└─────────────────────────────────────────────────┘
                      ↓
┌─────────────────────────────────────────────────┐
│              Coding Agent                        │
│  （开发者 - 循环迭代）                            │
│                                                  │
│  每轮开始：                                       │
│  1. 检查目录、读 git 记录、跑 init.sh            │
│  2. 从清单选一个未完成功能                       │
│  3. 执行并验证                                   │
│  4. 通过才标记完成并 commit                      │
└─────────────────────────────────────────────────┘
```

#### OpenClaw 的实践

虽然没有完全实现双 Agent，但我们通过 **AGENTS.md + MEMORY.md** 实现了类似效果：

```markdown
# AGENTS.md
## Every Session
1. Read `SOUL.md` — this is who you are
2. Read `USER.md` — this is who you're helping
3. Read `memory/YYYY-MM-DD.md` — recent context
4. 查询最近 10 条记忆 — user preferences

# MEMORY.md
## 当前进行中的项目
- openclaw-monitor: 已完成安全、性能优化
- 下一步：部署到生产环境
```

**核心思想**：每次会话开始，先"恢复记忆"，再继续工作。

### 支柱四：工作流引擎

**让 Agent 有"保障"**

#### Temporal 的威力

Temporal 提供了企业级的可靠性保障：

| 能力 | 说明 |
|------|------|
| **故障自愈** | 工作流崩溃后自动从断点恢复 |
| **智能重试** | 指数退避重试策略 |
| **人工在环** | 支持人工审核节点 |
| **超时处理** | 自动处理超时 |

#### 代码示例

```python
from temporalio import workflow

@workflow.defn
class ResearchWorkflow:
    @workflow.run
    async def run(self, query: str) -> str:
        # 步骤1：搜索
        results = await workflow.execute_activity(
            search_activity,
            query,
            start_to_close_timeout=timedelta(seconds=60)
        )
        
        # 步骤2：分析（如果失败会自动重试）
        analysis = await workflow.execute_activity(
            analyze_activity,
            results,
            start_to_close_timeout=timedelta(seconds=120),
            retry_policy=RetryPolicy(
                maximum_attempts=3,
                initial_interval=timedelta(seconds=10),
            )
        )
        
        # 步骤3：生成报告
        return await workflow.execute_activity(
            generate_report_activity,
            analysis,
            start_to_close_timeout=timedelta(seconds=60)
        )
```

## 最佳实践总结

### 1. 状态管理

```markdown
✅ 必须做：
- 使用 thread_id 管理会话
- 关键状态持久化到数据库
- 定期创建检查点

❌ 不要做：
- 依赖内存状态
- 不记录中间结果
- 没有恢复机制
```

### 2. 错误处理

```markdown
✅ 必须做：
- 分层错误处理（工具/Agent/系统）
- 可重试错误自动重试
- 记录详细错误日志

❌ 不要做：
- 单一 try-catch
- 静默忽略错误
- 无限重试
```

### 3. 架构设计

```markdown
✅ 必须做：
- 长任务拆分为可验证的迭代
- 使用监督者模式
- 每轮都验证进度

❌ 不要做：
- 单 Agent 自主完成长任务
- 不记录进度
- 依赖模型"记住"上下文
```

### 4. 监控告警

```markdown
✅ 必须做：
- 实时监控核心指标
- 设置告警阈值
- 完整日志审计

❌ 不要做：
- 无监控
- 无告警
- 无日志
```
