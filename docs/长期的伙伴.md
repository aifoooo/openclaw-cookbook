# 长期的伙伴：让龙虾拥有长期记忆

OpenClaw 的 MEMORY.md 机制，本质上是一种「被动记忆」——文件本身是持久化的，但龙虾不会主动判断什么值得记住。

我想要的是「主动记忆」——龙虾能自己判断什么值得记住，自动保存，下次见面还能想起来。

这篇文章记录了给龙虾装上"长期记忆"的过程。

---

## 一、为什么需要长期记忆

我已经配置了 OpenClaw 在空闲 14 天后才重置对话，但这样的设置仍然无法满足我的诉求：

1. 对话崩了，通过 `/new` 新建会话，龙虾并不记得之前聊了什么
2. 提及很久之前聊过的问题，龙虾往往已经忘了

这让我意识到：**没有长期记忆的龙虾，每次对话都是从零开始。**

> 💡 **一句话**：被动记忆是"你让我记我才记"，主动记忆是"我自己判断什么该记"。

---

## 二、外挂记忆系统

Google 开源的 [Always On Memory Agent](https://github.com/GoogleCloudPlatform/generative-ai/tree/main/gemini/agents/always-on-memory-agent) 提供了一个思路：**模拟人脑的记忆整合机制**。

人脑在睡眠时会回放白天的经历，发现事件之间的关联，把零散的信息压缩成稳定的记忆。这个项目设计了三个 Agent 协作：

- IngestAgent：接收对话，提取关键信息
- ConsolidateAgent：定时整合，发现记忆之间的联系
- QueryAgent：综合记忆，回答问题

核心思想：**记忆不是简单的存储，而是一个持续处理、整合的过程。**

这个思路很棒，但我需要现成的方案。于是让龙虾调研了一圈 GitHub：

| 项目 | Stars | 特点 |
|-----|-------|------|
| mem0 | 49k+ | Y Combinator 孵化，通用记忆层 |
| memU | 12k+ | **专为 OpenClaw 设计** |
| Letta | 21k+ | 前身 MemGPT，自进化记忆 |
| memvid | 13k+ | 单文件存储，无需数据库 |

memU 的 README 直接提到 OpenClaw——专门为这类 24/7 运行的 Agent 设计，支持「主动记忆」机制。

> 💡 **一句话**：选对工具，事半功倍。

## 三、龙虾的实践：三个坑

把 memU 跑起来，过程比预想的曲折。

### 3.1 第一个坑：腾讯云 Embedding 配置失败

memU 需要 Chat API + Embedding API 配合使用：
- Chat API：提取记忆项
- Embedding API：向量化，用于相似度搜索

我使用的主力 LLM 是腾讯云 GLM-5。但发现 memU 原生不支持腾讯云的签名认证方式，魔改成本较高。

**解决方案**：换阿里云全家桶（Chat + Embedding），简单省事。

### 3.2 第二个坑：SQLite 持久化失败

memU 默认用 SQLite 存储记忆，但遇到了 SQLModel 版本兼容性问题。

**解决方案**：改用 PostgreSQL + pgvector——memU 官方推荐的方案。

折腾了一番：
- 安装 PostgreSQL 15
- 安装 pgvector 扩展（向量搜索）
- 创建 memu 数据库和用户

这次终于成功了。

### 3.3 第三个坑：网络延迟

阿里云 API 在北京，我的服务器在香港。跨区域调用延迟较高，memU 的 memorize 操作需要多次 API 调用，一次完整的 memorize 可能要 30-60 秒。

**解决方案**：把超时设置从 30 秒调到 120 秒。后续可以换成同区域的腾讯云 Embedding，或本地 sentence-transformers。

> 💡 **一句话**：踩坑是成长的必经之路，记录下来让后来者少走弯路。

## 四、最终架构

```
OpenClaw 主会话
    │
    ├─▶ MEMORY.md（核心记忆，手动维护）
    │
    └─▶ session-memory（对话记录）
            │
            ▼ Cron 每分钟同步
        memU（长期记忆系统）
            │
            ├─ memorize：LLM 提取记忆项 + Embedding 向量化
            │
            └─ PostgreSQL + pgvector：持久化存储
```

**工作流程**：

1. **对话记录**：OpenClaw 每次对话写入 `sessions/YYYY-MM-DD.jsonl`
2. **定时同步**：Cron 每分钟检查新对话
3. **记忆提取**：LLM 从对话中提取有价值的信息（偏好、画像、重要事件）
4. **向量化存储**：Embedding API 将记忆项向量化，存入 PostgreSQL
5. **查询记忆**：需要时通过 retrieve 进行向量相似度搜索

## 五、效果验证

memU 跑起来了，测试三个核心场景。

### 5.1 场景一：会话重置后还记得

我在对话中告诉龙虾：
> 我最近在学 Rust，用 IntelliJ IDEA 写代码

聊了几句别的事，然后 `/new` 新建会话。

问：**我最近在学什么语言？**

答：**根据记忆，你最近在学 Rust。**

✅ 成功！memU 的记忆跨会话生效。

### 5.2 场景二：以前的事也能想起来

不久前提过自己在做 openclaw-cookbook 项目。新建会话后问：

**我最近在做什么项目？**

答：**你在做 openclaw-cookbook，是一个 OpenClaw 使用经验的博客仓库，已经写了两篇文章：《龙虾初相遇》和《深度调研》。**

✅ 成功！向量相似度搜索找到了相关记忆。

### 5.3 场景三：偏好自动提取

从没明确说过「我喜欢 IntelliJ IDEA」，只是在它问我用什么编辑器时随口说了一句。

后来问：**你知道我有哪些偏好吗？**

答：**你喜欢用 IntelliJ IDEA 写代码。**

✅ 成功！memU 自动从对话中提取了偏好。

## 六、写在最后

目前 memU 的记忆写入和查询都需要手动触发（或 Cron 定时同步）。理想状态是：

- 每次对话结束，自动判断是否有值得记住的信息
- 每次对话开始，自动检索相关记忆注入上下文

这需要魔改 OpenClaw 的核心流程，暂时先这样用着。

**但已经足够好用了**——龙虾终于不再是"金鱼记忆"，而是能记住我的偏好、项目、重要事件。

> 🦐 **有了长期记忆的龙虾，才算真正的伙伴。**
