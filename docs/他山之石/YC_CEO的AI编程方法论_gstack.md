# 他山之石：gstack —— YC CEO 的 AI 编程方法论

---

## 一、引子：一个人等于一支团队

2026年3月，Andrej Karpathy 在播客中说了一句话：

> "我大概从去年12月开始就没怎么手写代码了，这大概是职业生涯以来最大的转变。"

这话从 OpenAI 联合创始人口中说出来的分量，懂的都懂。

而 Y Combinator 的 CEO Garry Tan —— 这个管理着全球最大创业加速器、每天被无数创始人包围的人 —— 用另一种方式证明了这一点。

**他在 60 天内：**
- 产出 60 万行生产代码（其中 35% 是测试）
- 平均每天 1-2 万行代码
- 兼职完成（他同时还在全职运营 YC）

对比一下他自己在 2013 年发明的 Bookface（YC 内部社交网络）：那一年他提交了 772 次 commit。而 2026 年的前两个月，他已经有了 1237 次 commit。

同样的一个人，不同的时代。差距来自工具。

---

## 二、gstack 是什么

gstack 是 Garry Tan 把自己的 AI 编程方法论开源的项目。他的目标很直接：

> "让一个人能像二十个人的团队一样高效地开发软件。"

但 gstack 不是一个"AI 编程工具的集合"。Garry Tan 自己在文档里特别强调：

> **"gstack is a process, not a collection of tools."**

这不是一句营销话术。背后的逻辑非常清晰：

**AI 编程的核心瓶颈不是工具，而是流程。**

当你让 AI 直接写代码时，它会在毫秒之间跳过了所有设计决策，直接给你代码。但"理解要做什么"和"讨论如何构建"是两件完全不同的事。AI 把它们合并成了一件。

gstack 提供的是一套工程流程，让 AI 在每个阶段做该做的事。

---

## 三、核心流程：Think → Plan → Build → Review → Test → Ship → Reflect

gstack 的每个技能不是孤立的，而是形成了一个完整的工程周期：

```
┌─────────────────────────────────────────────────────────────────┐
│                                                                 │
│  /office-hours    →  重新思考产品定义                            │
│        ↓                                                     │
│  /plan-ceo-review →  CEO/Founder 评审产品方向                    │
│        ↓                                                     │
│  /plan-eng-review →  工程经理锁定架构和实施方案                  │
│        ↓                                                     │
│  /review          →  高级工程师代码审查                          │
│        ↓                                                     │
│  /qa              →  QA 真实浏览器测试                          │
│        ↓                                                     │
│  /ship            →  合并代码、创建 PR                         │
│        ↓                                                     │
│  /retro           →  团队反思、回顾改进                        │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

每个技能的输出，会成为下一个技能的输入。

比如 /office-hours 会生成一份设计文档，这份文档会自动流入 /plan-ceo-review 和 /plan-eng-review。/plan-eng-review 写的测试计划，会被 /qa 自动拾取。整个流程不需要复制粘贴。

## 四、28 个技能详解

gstack 包含 28 个技能，Garry Tan 把它们组织成七类角色。

### 4.1 产品与战略层

| 技能 | 角色 | 职责 |
|------|------|------|
| `/office-hours` | YC Office Hours | 六个追问重新定义产品，在写代码之前想清楚到底在做什么 |
| `/plan-ceo-review` | CEO / Founder | 寻找"藏在需求里的10分产品"，四种模式：扩展/选择性扩展/保持范围/缩减 |
| `/plan-eng-review` | 工程经理 | 锁定架构、数据流、状态机、边界情况、测试计划 |
| `/plan-design-review` | 高级设计师 | 对设计的每个维度打分(0-10)，解释10分是什么样的，然后修改计划达到10分 |

**最有趣的是 /office-hours 的工作方式。**

Garry Tan 分享了一个真实案例：有人来说"我想做一个日历每日简报应用"。然后 YC 风格的追问开始了 —— 不是问功能，而是问痛苦。

用户描述了真实的痛苦：多个 Google 日历信息陈旧、地点错误、准备时间太长、结果还不满意。

AI 的回应是：

> "我要挑战你的框架。你说的是'日历简报应用'，但你实际描述的是一个**个人首席运营官 AI**。"

然后它列出了用户自己都没意识到的五个能力：
- 跨账号监测日历，检测过时信息、缺失地点、权限缺口
- 生成真实的准备工作 —— 不是物流摘要，而是为董事会会议、播客、资金募集准备的真正智力工作
- 管理 CRM —— 联系谁、关系如何、各自要什么、历史是什么
- 优先排序时间 —— 标记何时需要提前开始准备、按重要性排序事件
- 以杠杆换时间 —— 主动寻找可以委托或自动化的方式

用户本来要做一个日历应用，结果变成了一个价值高十倍的东西。

**因为它倾听了用户的痛苦，而不是字面需求。**

### 4.2 工程与代码层

| 技能 | 角色 | 职责 |
|------|------|------|
| `/review` | 高级工程师 | 找出那些能通过 CI 但会在生产环境爆掉的 bug。自动修复明显的，标记需要判断的 |
| `/investigate` | 调试专家 | 系统性根因调试。"铁律"：没有调查就没有修复。追踪数据流、测试假设、三次失败后停止 |
| `/design-review` | 会写代码的设计师 | 对线上站点做80项视觉审查，然后修复发现的问题。每次修复原子提交，有 before/after 截图 |

`/review` 特别值得注意。它的核心逻辑是：**那些通过测试但在生产环境爆掉的 bug**。

比如：
- N+1 查询
- 过期读取
- 竞态条件
- 错误的信任边界
- 缺失的索引
- 逃逸的 bug
- 错误的重试逻辑
- 被遗忘的枚举处理器 —— 新增一个状态时，/review 会追踪它在整个代码库中的每个 switch 语句

自动修复明显的机械问题（死代码、过时注释、N+1查询），但把设计决策类的问题标记出来让人类判断。

### 4.3 测试与质量层

| 技能 | 角色 | 职责 |
|------|------|------|
| `/qa` | QA 负责人 | 用真实浏览器测试、找 bug、用原子提交修复、重新验证。每次修复自动生成回归测试 |
| `/qa-only` | QA 报告者 | 和 /qa 一样的方法论，但只报告不修改代码 |
| `/browse` | QA 工程师 | 给 AI 一双眼睛。真实 Chromium 浏览器、真实点击、真实截图，每条命令约100ms |

`/qa` 的工作方式很巧妙：当你处于一个 feature branch 时，只需要说 `/qa` —— 它会读取你的 git diff，自动识别你的改动影响了哪些页面和路由，然后针对这些页面进行测试。不需要提供 URL，不需要手动写测试计划。

四种模式：
- **Diff-aware**（feature branch 自动开启）—— 读取 main 的 diff，专门测试受影响页面
- **Full** —— 系统性探索整个应用，5-15分钟，产出5-10个有据可查的问题
- **Quick** —— 30秒冒烟测试
- **Regression** —— 运行完整测试后与之前基准对比

### 4.4 设计与创意层

| 技能 | 角色 | 职责 |
|------|------|------|
| `/plan-design-review` | 高级设计师 | 在实现前审查计划，七轮审查：信息架构、交互状态覆盖、用户旅程、AI 垃圾风险、设计系统对齐、响应式/可访问性、未解决的设计决策 |
| `/design-consultation` | 设计伙伴 | 从零开始建立完整设计系统。研究竞品、提出创意风险、生成逼真的产品 mockup |
| `/design-review` | 会写代码的设计师 | 线上站点的视觉审查 + 修复循环 |

`/design-consultation` 的工作流程很有意思。当你什么都没有时（没有设计系统、没有字体选择、没有配色方案），它会：

1. 先问你的产品和用户
2. 研究竞品 landscapes（真的去浏览竞品网站截图分析）
3. 提出三套字体、三套配色方案，并告诉你哪些是安全选择（符合类别预期）、哪些是创意风险（让你脱颖而出）
4. 生成交互式 HTML 预览 —— 不是色板和字体样本，而是真实的产品页面
5. 写 DESIGN.md 和更新 CLAUDE.md，让未来所有 AI session 都遵循这个系统

### 4.5 安全与合规层

| 技能 | 角色 | 职责 |
|------|------|------|
| `/cso` | 首席安全官 | OWASP Top 10 + STRIDE 威胁模型审计。扫描注入、认证问题、加密、访问控制。17个已知的误报排除规则 |

### 4.6 发布与运维层

| 技能 | 角色 | 职责 |
|------|------|------|
| `/ship` | 发布工程师 | 同步 main、运行测试、审计覆盖率、推送、创建 PR。如果项目没有测试框架，自动引导搭建 |
| `/land-and-deploy` | 发布工程师 | 合并 PR、等待 CI 和部署、验证生产环境健康。从"已批准"到"生产验证"一步完成 |
| `/canary` | SRE | 部署后监控循环。监测控制台错误、性能回归、页面失败 |
| `/benchmark` | 性能工程师 | 基线页面加载时间、Core Web Vitals、资源大小。比较每次 PR 的前后差异 |
| `/document-release` | 技术写作 | 更新所有项目文档以匹配你刚发布的内容。自动捕捉过时的 README |

`/ship` 有一个细节很值得注意：**它会在创建 PR 前检查评审就绪状态**。如果工程评审缺失，它会询问你是否继续。

这解决了一个真实问题：很多分支死于"有趣的开发工作已完成，只剩无聊的发布工作"——人类会拖延这部分，但 AI 不应该。

### 4.7 安全护栏

| 技能 | 角色 | 职责 |
|------|------|------|
| `/careful` | 安全护栏 | 危险命令警告（rm -rf、DROP TABLE、force-push）。说"be careful"激活，可覆盖任何警告 |
| `/freeze` | 编辑锁 | 将文件编辑限制在单一目录，防止调试时意外修改范围外文件 |
| `/guard` | 全安全 | /careful + /freeze 合二为一，适合生产环境工作 |
| `/unfreeze` | 解锁 | 移除 /freeze 边界 |

### 4.8 团队与协作层

| 技能 | 角色 | 职责 |
|------|------|------|
| `/retro` | 工程经理 | 团队感知的每周回顾。按人分析产出、代码提交连续性、测试健康趋势、成长机会。`/retro global` 跨所有项目运行 |
| `/codex` | 第二意见 | OpenAI Codex CLI 的独立代码审查。三种模式：审查（通过/失败门控）、对抗性挑战、开放咨询 |

## 五、三大核心原则

gstack 不只是一套技能，它还有一套哲学。这些原则被注入到每个 workflow skill 的前言中。

### 5.1 Boil the Lake（煮湖）

> **"完成度是廉价的。"**

Garry Tan 给出了一张压缩比表格：

| 任务类型 | 人类团队 | AI 辅助 | 压缩比 |
|---------|---------|---------|--------|
| 样板代码/脚手架 | 2天 | 15分钟 | ~100x |
| 测试编写 | 1天 | 15分钟 | ~50x |
| 功能实现 | 1周 | 30分钟 | ~30x |
| Bug修复+回归测试 | 4小时 | 15分钟 | ~20x |
| 架构/设计 | 2天 | 4小时 | ~5x |
| 研究/探索 | 1天 | 3小时 | ~3x |

**这张表格改变了一切关于"做还是不做"的决策。**

以前团队会跳过"最后的10%完整性"，因为成本太高。现在这10%的成本只有几秒钟。

所以：**当完整实现比走捷径只多70行代码时，选择完整实现。** 70行的差距用 AI 只需要几秒钟。"走捷径"是遗留思维，因为那时候人类工程时间是瓶颈。

Boil the Lake 的反模式：
- "选B方案吧 —— 覆盖90%但代码更少。"（如果A方案只多70行，选A）
- "测试留到后续 PR 吧。"（测试是成本最低的湖，完全值得煮）
- "这要两周。"（改成："人类两周 / AI辅助约一小时"）

### 5.2 Search Before Building（建前先搜）

> **"1000x 工程师的第一反应是'有人解决过这个问题吗？'而不是'让我从头设计。'"**

在构建任何涉及不熟悉模式、基础设施或运行时能力的东西之前 —— 停下来，先搜索。

检查的成本接近于零。不检查的成本是你在重新发明一个更差的东西。

三层知识来源：

**Layer 1: Tried and true** —— 标准模式、经过实战验证的方法、深度分布的东西。你可能已经知道了，风险不在于不知道 —— 而是你以为显而易见的答案偶尔是错的。检查成本接近于零，而偶尔质疑"常识"可能是你做出惊人发现的地方。

**Layer 2: New and popular** —— 当前最佳实践、博客文章、生态系统趋势。去搜索，但仔细审查 —— 人类容易狂热。人群对新事物的判断既可能过于恐惧，也可能过于贪婪。

**Layer 3: First principles** —— 从手头具体问题推理得出的原创观察。这是三者中最有价值的。重视它超过一切。

搜索的真正价值不在于找到一个可复制的解决方案，而在于：
- 理解每个人在做什么以及为什么（Layer 1+2）
- 对他们的假设进行第一性原理推理（Layer 3）
- 发现常识方法在哪个具体点上错了

**这才是11分的产品。**

### 5.3 做自己的问题

> **"最好的工具解决的是你自己的问题。"**

gstack 存在是因为它的创造者需要它。每个功能都是因为需要而构建，而不是因为被要求。

如果你为自己构建东西，相信这个直觉：**真实问题的具体性永远打败假设的通用性。**

## 六、对 AI 编程工程的启发

### 6.1 流程 > 工具

这是最核心的教训。

gstack 的本质不是 28 个技能，而是这 28 个技能如何串联成流程。每个技能的输出是下一个的输入，每个阶段有明确的"完成标准"。

这与 Martin Fowler 的 Design-First 方法论一脉相承：先讨论，再实现。gstack 把这个原则工程化了 —— 不是讨论一次，而是每个阶段都讨论，每个阶段的结论都流入下一个阶段。

### 6.2 AI 编程的规模效应

Garry Tan 展示了 AI 编程的真正规模效应：

- 样板代码：100x 压缩
- 测试编写：50x 压缩
- 功能实现：30x 压缩

这意味着**以前需要团队分工才能完成的完整度，现在一个人加 AI 就能做到**。这不只是效率提升，而是**工程组织的根本性重构**。

### 6.3 人的角色转变

当 AI 处理了几乎所有"写代码"的工作后，人类做什么？

gstack 的答案是：**品味、判断、愿意做完整的事。**

具体来说：
- 定义正确的问题（/office-hours）
- 做设计决策（/plan-ceo-review、/plan-eng-review）
- 判断而非执行
- 在关键时刻介入，而非全程参与

这不是人被替代了，而是人被升级了 —— 从代码执行者变成问题定义者和决策者。

### 6.4 安全融入流程

gstack 有一个被很多人忽视的特点：它把安全做成了流程的一部分，而非事后补救。

`/careful`、`/freeze`、`/guard` 这些技能不是"安全插件"，而是工程流程的自然组成部分。当你需要调试时，冻结范围防止意外修改；当你处理生产数据时，开启全安全模式。

## 七、与 cookbook 中已有内容的关联

在 cookbook 的"如何让AI写出杰出的软件"文章中，我们探讨了 Design-First、Spec-Driven Development 等方法论。gstack 提供了这些方法论的**完整工程实现**：

| 文章概念 | gstack 实践 |
|---------|------------|
| Design-First 五层模型 | /office-hours → /plan-ceo-review → /plan-eng-review → /review → /qa → /ship |
| Spec-Driven Development | 每个技能产出结构化文档，流入下一个技能 |
| Vibe Engineering（Simon Willison） | 强调工程实践（测试、代码审查）放大开发者专业能力 |
| Human-in-the-Loop | 每个阶段都有 checkpoint，人类批准后才进入下一阶段 |

同时，gstack 的三大原则（Boil the Lake、Search Before Building、做自己的问题）与 cookbook 中强调的**工程化思维**高度吻合。

## 八、思考与行动

### 值得借鉴的

1. **建立环环相扣的工程流程**：不是一堆工具，而是一个从思考到反思的完整周期
2. **每个阶段有明确的输入输出**：文档作为各阶段之间的桥梁
3. **把安全融入流程**：不是事后打补丁
4. **重视"完成度"的成本**：AI 让完整实现的成本趋近于零
5. **Search Before Building**：先理解 landscape，再决定构建什么

### 需要注意的

1. **gstack 目前绑定 Claude Code**：虽然说支持其他 agent，但实践上对 Claude Code 的依赖很深
2. **全面采用需要较大调整**：对于习惯了自己工作流的团队，引入整套流程有学习成本
3. **中文资料匮乏**：目前主要是英文文档和英文视频讲解

## 九、结语

Garry Tan 在 gstack 的 README 中写了一句话：

> "The engineering barrier is gone. What remains is taste, judgment, and the willingness to do the complete thing."

工程障碍已经消失了。剩下的只有品味、判断，和做完整的事的意愿。

这不是说工程不重要了。而是**工程能力的表现形式变了** —— 从"能写代码"变成"能定义正确的问题、能在每个阶段做出正确的决策、能让 AI 产出完整的实现"。

一个人等于一支团队的时代已经来了。工具已经有了。流程正在成熟。

**剩下的，是我们自己准备好了吗？**
