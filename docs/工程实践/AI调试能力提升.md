# AI 调试能力提升

龙虾调试代码的效率远不如人类工程师。这篇笔记记录了原因分析和改进方向。

---

## 一、问题：AI 调试效率低下

### 表现

| 问题 | 说明 |
|------|------|
| **线性尝试** | 改代码 → 再改 → 还不行 → 继续改，陷入循环 |
| **忽视日志** | 后端日志明明显示问题，却在改前端代码 |
| **缺乏假设** | 没有系统性地建立和验证假设 |
| **重复错误** | 同样的错误模式反复出现 |

### 根本原因

#### 1. 上下文腐烂

随着对话深入，历史消息、工具结果、环境状态信息累积，Token 呈指数级增长。

**案例**：一个被分解为 10 个子任务的任务会产生 41 条新增上下文记录。

**后果**：
- 上下文超限时截断历史 → "失忆"
- 关键信息被淹没在噪音中
- 推理质量下降

#### 2. 推理模式差异

| 维度 | AI Agent | 人类工程师 |
|------|---------|-----------|
| **推理模式** | 模式匹配，缺乏深层理解 | 假设驱动，基于经验推理 |
| **调试方法** | 线性尝试，容易循环 | 系统性排查（二分法、日志分析） |
| **知识应用** | 上下文限制，易遗忘 | 长期记忆，经验复用 |
| **决策能力** | 缺乏判断力，盲目执行 | 战略判断，风险评估 |

## 二、人类优秀调试工程师的方法论

### 核心流程

```
建立假设 → 设计实验 → 验证假设 → 精准修复
```

### 关键方法

| 方法 | 说明 | 示例 |
|------|------|------|
| **二分法调试** | 缩小问题范围，快速定位 | 注释一半代码，看错误是否消失 |
| **日志分析** | 从海量日志中提取关键信息 | grep 错误关键字，查看调用栈 |
| **假设验证** | 提出假设 → 设计实验 → 验证结果 | "可能是 WebSocket 连接问题" → 检查连接状态 |
| **经验复用** | 类似问题的解决方案迁移 | 这个错误上次是配置问题，先查配置 |

### 调试思维清单

```markdown
【第一步】收集信息
- 错误信息是什么？
- 日志里有什么线索？
- 最近改了什么？

【第二步】建立假设
- 最可能的原因是什么？（优先级排序）
- 如何验证这个假设？

【第三步】验证假设
- 设计最小实验验证
- 一次只验证一个假设
- 记录实验结果

【第四步】精准修复
- 确认根因后，只改必要的代码
- 修复后验证问题是否解决
- 记录解决方案
```

## 三、提升 AI 调试能力的方案

### 1. 上下文工程七大策略

| 策略 | 说明 | 实践 |
|-----|-----|-----|
| **压缩** | LLM 驱动语义压缩，保留关键信息 | 用一句话总结 100 行日志 |
| **替换** | 用引用替换大段内容 | "详见日志文件 /tmp/xxx" |
| **保留** | 保留最近对话作为"热缓存" | 最近 5 轮对话不压缩 |
| **锚定** | 设置关键信息锚点，防止丢失 | "重要：WebSocket 连接失败" |
| **合并** | 合并相似信息，减少冗余 | 合并重复的错误日志 |
| **共享** | 多 Agent 间共享内存或消息传递 | 使用共享的 MEMORY.md |
| **动态扩展** | 工具按需加载，避免"降智" | 只加载当前任务需要的 skill |

### 2. 设计可调试的 Agent

**在 Agent 的系统提示中添加**：

```markdown
【调试模式】
每次工具调用前，输出：
1. [思考] 我在想什么
2. [假设] 我怀疑的原因是什么
3. [实验] 我打算如何验证

每次工具调用后，输出：
1. [结果] 工具返回了什么
2. [结论] 验证了我的假设吗？
3. [下一步] 接下来要做什么

【防循环机制】
- 同一工具最多连续调用 3 次
- 超过限制后必须：停止并报告问题
- 记录调用历史，避免重复

【异常处理】
- 工具返回错误 → 重试一次，仍失败则报告
- 结果不符合预期 → 尝试其他方法
- 无法完成任务 → 说明原因，请求帮助
```

### 3. 自动化失败归因

**MAST 故障分类法**（来自学术论文）：

| 故障类型 | 占比 | 说明 |
|---------|------|------|
| **规范问题** | 41.76% | 任务描述不清、目标不明确 |
| **智能体间错位** | 36.94% | 多 Agent 协作时信息不同步 |
| **任务验证** | 21.30% | 结果验证机制缺失 |

**应对策略**：
- 规范问题 → 明确任务描述，提供示例
- 智能体间错位 → 使用共享内存，定期同步
- 任务验证 → 添加验证步骤，检查结果

### 4. 人机协同调试模式

| 策略 | 说明 |
|-----|-----|
| **设定边界** | 为 AI 任务设定时间限制（如 30 分钟） |
| **接受不完美** | AI 70% 可用输出，剩余人类补全 |
| **分离思考与执行** | 保留无 AI 的深度思考时间 |
| **明确分工** | AI 做探索和生成，人类做判断和验证 |

## 四、实战案例

### 案例 1：WebSocket 连接失败

**错误做法（AI 典型表现）**：
```
1. 修改前端代码添加日志
2. 重启前端服务
3. 修改后端配置
4. 重启后端服务
5. 还是失败，继续改代码...
```

**正确做法（人类工程师）**：
```
1. 查看后端日志 → 发现 "Broadcasting to 1 connections"
2. 查看前端日志 → 没有 WebSocket 连接日志
3. 建立假设：前端 WebSocket 代码没有执行
4. 验证假设：检查前端 onMounted 是否调用
5. 发现问题：环境变量配置错误
6. 精准修复：只修改环境变量
```

### 案例 2：API 返回空数组

**AI 的表现**：
- 立即修改前端代码
- 添加防闪烁逻辑
- 改了好几处，问题依然存在

**更好的做法**：
1. **先看后端日志**：API 实际返回了什么？
2. **检查前端请求**：请求参数对吗？Token 正确吗？
3. **建立假设**：可能是认证问题或网络问题
4. **验证假设**：用 curl 直接测试 API
5. **精准修复**：只改问题点

## 五、核心结论

### 关键洞察

1. **上下文工程是基石**：实施系统化的上下文管理，解决"上下文腐烂"和记忆缺失问题

2. **可调试性需内建**：从设计之初就为 Agent 植入结构化思考输出、调用日志和防护机制

3. **重视系统设计**：AI Agent 成功 70% 取决于清晰的规范、良好的协作设计和可靠的验证机制

4. **重新定义人机分工**：
   - AI 擅长：快速生成、探索、执行
   - 人类擅长：深度理解、战略判断、最终验证

5. **投资基础设施**：关注上下文效率、Agent 授权、审计轨迹、运行时安全等长期存在的问题

### 最佳实践

```markdown
【调试前】
1. 查看日志（后端、前端、系统）
2. 收集错误信息
3. 确认最近改了什么

【调试中】
1. 建立假设（优先级排序）
2. 设计最小实验验证
3. 一次只验证一个假设
4. 记录实验结果

【调试后】
1. 确认根因
2. 精准修复（只改必要的代码）
3. 记录解决方案到 MEMORY.md
4. 总结经验教训
```
